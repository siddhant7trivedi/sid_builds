# Retrospectives Skill

## Purpose
Extract durable learning from what just happened — and turn it into concrete action, not just a feelings session.

---

## Core Principles

1. **Blameless, not blame-free** — Name what went wrong systemically; don't scapegoat individuals
2. **Actions must be actionable** — "Communicate better" is not an action; "PM will send a weekly status by Friday noon" is
3. **Retrospectives are investments** — They only pay off if action items are tracked and closed
4. **Different formats for different purposes** — Sprint retro ≠ launch retro ≠ incident review
5. **Celebrate what worked** — Learning only from failure is a recipe for burnout

---

## Retrospective Types

| Type | When | Duration | Participants |
|------|------|----------|-------------|
| Sprint Retro | End of each sprint | 60–90 min | Squad |
| Launch Retro | 2–4 weeks post-launch | 90 min | Cross-functional |
| Quarterly Product Review | End of quarter | 2–3 hours | PM + Eng + Design + Exec |
| Experiment Post-Mortem | After any significant experiment | 60 min | PM + Data + Eng |
| Incident Review | After any P0/P1 incident | 2 hours | Eng + PM + CS |
| Strategy Retro | Annual | Half day | Leadership |

## Retrospective Formats

### Start / Stop / Continue
Simple and effective for sprint retros:
- **Start**: What should we begin doing?
- **Stop**: What should we stop doing?
- **Continue**: What's working that we should keep?

### 4 Ls
Good for launch or quarterly:
- **Liked**: What did we enjoy or value?
- **Learned**: What did we learn?
- **Lacked**: What was missing?
- **Longed For**: What do we wish we had?

### Plus/Delta
Quick and positive-framing:
- **Plus (+)**: What worked well?
- **Delta (Δ)**: What should change?

### 5 Whys (Root Cause)
For incident or failure retros:
- Take the primary issue
- Ask "why" 5 times to get to root cause
- Don't stop at symptoms

### Sailboat / Speedboat
Visual metaphor for strategy retros:
- **Wind** (what's pushing us forward)
- **Anchor** (what's slowing us down)
- **Island** (our goal)
- **Rocks** (risks ahead)

## PM-Led Retrospective: Product Outcome Review

Beyond process, PMs should retro on outcomes:
- Did we achieve our OKRs?
- Which bets paid off? Which didn't?
- Which assumptions were wrong?
- What would we do differently?
- What should we carry forward into the next cycle?

---

## Facilitation Principles

1. **Time-box each section** — Don't let one topic eat the whole session
2. **Everyone writes before anyone speaks** — Silent writing prevents groupthink
3. **Vote on themes** — Use dot voting to surface top items before discussion
4. **End with actions, not insights** — The last 15 minutes must produce action items
5. **PM documents, not just facilitates** — Capture in real-time; don't rely on memory

---

## Action Item Quality Standard

Every action item must have:
- **Owner**: One named person (not "the team")
- **Action**: Specific, doable task (not a vague principle)
- **Due date**: Explicit deadline

**Bad action item**: "Improve communication between PM and Eng"
**Good action item**: "PM to share sprint goals doc every Monday by 9am starting next sprint — Owner: [PM Name] — Due: Next sprint Day 1"

---

## Patterns to Watch For

| Pattern | Signal | Question to Ask |
|---------|--------|-----------------|
| Same issues repeat | Actions not closing | Are we actually following through? |
| Retros too long | No time-boxing | Shorten each section; hard stops |
| Same people dominate | Psychological safety issue | Use async writing first |
| No one raises real issues | Trust problem | Run anonymous async retro |
| Actions never reviewed | No accountability | Start every retro by reviewing last retro's actions |

---

## Quality Checks

- [ ] Retro started by reviewing previous action items
- [ ] All participants had a chance to contribute
- [ ] Root causes surfaced (not just symptoms)
- [ ] Every action item has an owner, action, and due date
- [ ] Actions documented in a shared, accessible place
- [ ] Follow-up retro scheduled (for incident reviews)
- [ ] Celebration of wins was included
